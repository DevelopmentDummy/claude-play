# Claude Play 개선 감사 백로그 (2026-05-30)

> 멀티 에이전트 감사: 58 agents, 48건 발굴 → **39건 확정** (회의적 교차검증 통과). 자동 생성 산출물.

## 최우선 추천 (synthesis)

Spec the Antigravity runtime-prompt persistence fix first ("Antigravity loses the ENTIRE runtime prompt on every resume/reopen"). It is the single highest-leverage item: Antigravity is a first-class RP provider, and today its entire immersion ruleset — the <dialog_response>/<choice>/<break/> format spec, Korean-native style rules, responseLength/innerMonologue/generateChoices options, and the panel-action [AVAILABLE] spec — is delivered ONLY via the new-cascade --prompt-interactive arg, where it is hard-capped at MAX_PRIMER_CHARS=28000 while session-shared.md (~31.7KB) + primer + panel markdown is ~37.5KB (silent tail + panel-action truncation from turn 1), and the resume branch (antigravity-process.ts:92) re-delivers none of it. I verified the three load-bearing facts directly: the resume PowerShell branch passes only --conversation/--dangerously-skip-permissions with no system prompt; writeAntigravityInstructions (session-manager.ts:2250) writes persona-only GEMINI.md and takes no runtime-prompt arg; and the contrast pattern already exists for Gemini/Kimi (writeKimiInstructions(projectDir, runtimePrompt) at 2268), so the fix is a low-risk mirror, not a new mechanism. Leverage: this finding is also the explicit prerequisite for the options-restart finding (rank 20) and is thematically paired with the turn-header drift (rank 2), so fixing it unblocks/de-risks two other RP items. Risk is contained and the verdict already specifies the one runtime assumption to validate before implementing — that agy re-reads GEMINI.md on --conversation resume — verifiable via the brain transcript at ~/.gemini/antigravity-cli/brain/{cascadeId}/.system_generated/logs/transcript.jsonl, with a system-event-prepend fallback if it does not. Frame the spec around the verifiable active defect (silent truncation + non-persistent rules) rather than "zero rules on resume," keep a short first-turn primer stub so the auto-cascade still kicks off, and reduce --prompt-interactive to that stub once the full ruleset lives in GEMINI.md.

## 테마

### [RP 경험] Antigravity runtime-prompt integrity & turn assembly

The Antigravity provider silently loses or truncates its immersion ruleset (session-shared.md format spec, style rules, response/choice options, panel-action [AVAILABLE]) on resume/reopen/options-restart, and the primary user-message path drops the per-turn orchestrator role-reanchor that exists specifically to stop Antigravity hallucinating a supervisor. These are the same provider, the same delivery-of-rules failure mode, and one fix (persist runtime prompt to GEMINI.md like Gemini/Kimi) is an explicit prerequisite for the options-restart fix.

### [교차] Session-state write integrity (silent loss, no atomicity, lossy merge)

Every path that mutates RP state (custom tool results, on-message/on-assistant/style-check hooks, panel writes) does a non-atomic shallow read-modify-write with bare empty catches: failures are reported as success, there is no delete semantics, nested objects are clobbered, and a crash mid-write corrupts JSON that buildJsonLint then flags. Centralizing this with atomic tmp+rename + retryOnWindowsLock + logging fixes a real authoring/robustness gap.

### [코드 건전성] System-file / config single-source-of-truth drift

Constants that partition 'system' files from persona data, and endpoint/port defaults, are hand-redefined many times and DISAGREE. The SYSTEM_JSON exclusion set is duplicated 8x across 4 files with inconsistent contents (same directory yields different data/system views to hooks vs panels vs persona-info), and the self-API/GPU-manager/TTS port URLs are reconstructed as scattered magic strings. One exported constant/helper per concept removes the drift class of bug.

### [코드 건전성] Provider runtime duplication & the exit-event contract gap

The four CLI providers copy-paste the StringDecoder/flush/NDJSON stdout scaffold, the clean-env block, and (Codex+Kimi) the full JSON-RPC request/response/timeout machinery; meanwhile Codex never emits the shared 'exit' event so a mid-turn crash leaves idle waiters hanging. The exit-event gap is a one-line correctness fix; the scaffold/JSON-RPC dedupes are opportunistic DRY with modest net value given the genuinely-divergent parse/semantics and no test harness.

### [코드 건전성] God-class / mixed-responsibility extractions

Three oversized modules bundle unrelated subsystems: SessionManager (2424 lines: path-safety, persona/profile/session CRUD, sync/diff engine, runtime-config writers, system-prompt assembly, plus ~11 near-identical provider sessionId getter/setters and a latent missing-antigravity-branch bug in the builder variants), session-instance.ts (a ~280-line in-class 3-provider TTS engine, plus 4 hook runners re-implementing the same context loader), and comfyui-client.ts (HTTP client fused with ~700 lines of pure graph-surgery). Mostly large-effort, navigability-only wins in a no-test single-user service; the sessionId-getter dedup is the cheap exception and also fixes the builder bug.

### [프론트엔드/UX] Lobby & onboarding flow failure handling

First-touch flows swallow failures and strand the user: session start closes the modal then silently does nothing on a non-2xx; deletions are non-optimistic-or-non-reverting with no feedback; the required first-profile dialog closes even when the save POST failed (phantom profile dismisses the gate); the setup wizard dead-ends on a manual-restart screen despite an existing restart orchestrator; and PersonaStartModal offers a 'No profile' option it then hard-disables. Common fix: await→check res.ok→surface error/keep modal open, plus mount ToastEffect on the lobby.

### [프론트엔드/UX] Accessibility & interaction-affordance polish

A cluster of low-urgency a11y/interaction gaps for this single-user app: no prefers-reduced-motion handling, missing aria-labels on the chat textarea and icon-only buttons, a byte-identical voice-chat toggle glyph for on/off (color-only state, a real copy-paste bug), clickable non-focusable divs in ToolBlock/InlineImage, a few low-contrast interactive labels, no empty-state for zero-message sessions, and a mouse-only resize handle.

### [프론트엔드/UX] Panel lifecycle & overlay correctness

Panel-system correctness papercuts: DockPanel/ModalPanel run panel scripts without PanelSlot's sandbox so re-renders leak window listeners/timers over a long session; ImageModal's z-index loses to stacked/full-screen modals so zoom appears behind its own modal; the topmost ModalPanel re-wraps the global bridge.sendMessage without cleanup; modals lack dialog semantics/initial focus; and shared tool panels auto-mount into every session with no watcher and a flat action namespace (cross-persona contamination).

### [RP 경험] Style-check & skill-propagation correctness

The style-check reviewer and the copied skill files have correctness drift: the prose reviewer is fed RAW assistant output (choice JSON, $IMAGE/$PANEL tokens, dialog wrappers) which corrupts the very italic/dash/length counts it exists to produce; several image SKILL.md curl examples hardcode localhost:3340 defeating {{PORT}} substitution; and a stale duplicate generate-image-gemini skill under comfyui/ collides by basename with the canonical copy and only wins-or-loses by FS iteration order.

### [RP 경험] Panel-action availability authority

available_when / [AVAILABLE] is evaluated only client-side against possibly-stale panelData (one render behind the engine write), has no server-side enforcement, and silently returns false on any eval error so a typo permanently hides an action with zero author feedback. Latent today (no live persona uses available_when), so only the cheap dev-warning + freshest-snapshot read are worth doing now.

### Turn-state header is assembled in two divergent code paths that drift

- **차원**: RP 경험 · **위치**: `src/lib/session-instance.ts:422-434 vs src/app/api/chat/send/route.ts:35-41` · **임팩트/노력(조정)**: medium/small · **확신**: high
- **문제**: The [STATE]-equivalent prefix block (event headers, JSON lint, hint snapshot, action log, + user text) is built independently in two places. `sendMessage()` (server-initiated turns: scheduler ticks, fire-ai notifications, silent retries, restart-recovery) ALSO prepends the antigravity `orchestratorReminder` and resets `silentRetryDone`/waits for idle. The `/api/chat/send` route (the PRIMARY user path) deliberately bypasses sendMessage and re-implements the same assembly WITHOUT the orchestratorReminder and without the silentRetry reset. So a real antigravity user message never gets the role-reanchor that the codebase added specifically to stop antigravity from hallucinating a supervisor — only server-injected turns do. Any future header added to one path silently won't apply to the other, and the ordering/含 of parts is duplicated by hand.
- **제안(검증반영)**: Accept the finding, with a sharper, side-effect-aware fix. Add ONE method on SessionInstance, e.g. `buildTurnPrefix(text: string, { isOOC }: { isOOC: boolean }): string` that internally: (1) computes orchestratorReminder = this._provider === "antigravity" ? ROLE_REANCHOR : "" (move the literal to a module const so it cannot drift), (2) gates the flushing parts on isOOC EXACTLY as the route does today — `const eventHeaders = isOOC ? "" : this.flushEvents()`, same for hintSnapshot/actionHistory — so the queue-clearing side effects stay gated and pending events are preserved across OOC turns, (3) always includes jsonLint + text, (4) returns parts.join("\n"). Then: sendMessage() calls `this.buildTurnPrefix(text, { isOOC: false })` (its callers are never OOC, so behavior is unchanged except the reminder now flows through the single source) and the route calls `instance.buildTurnPrefix(text, { isOOC })` then `instance.claude.send(...)`, keeping its own waitForReady/503/addUserToHistory/pendingToolUseId logic untouched. Net effect: antigravity user messages finally get the L2 reanchor the commit intended; the other 4 providers are unaffected (empty string). SEPARATELY (do NOT bundle, but worth flagging to the author): the route should also reset silentRetryDone on a real user turn so the "max 1 retry per user input" guarantee actually holds — expose a tiny `instance.beginUserTurn()` or have buildTurnPrefix reset it when called for a non-silent-retry user turn. Verification before merge: run one antigravity session, send a normal RP message, and confirm via antigravity-stream.log / brain transcript that the [ROLE] header now precedes the user text and the supervisor-hallucination meta-leak no longer recurs on a ComfyUI-async turn. Keep the change behind no flag — it is a pure correction toward documented intent.
- **위험**: Adding orchestratorReminder to the user path changes the prompt antigravity users see — could shift behavior; verify with an antigravity session. OOC gating differs between the two (route drops everything for OOC, sendMessage doesn't) so the shared helper must take isOOC and preserve each caller's current gating.

### Tool-action data writes silently swallow failures and still report success

- **차원**: 코드 건전성 · **위치**: `src/app/api/sessions/[id]/tools/[name]/route.ts:163-164, 174-181` · **임팩트/노력(조정)**: medium/small · **확신**: high
- **문제**: When a persona tool action returns variable/data patches, the writes to variables.json and to each data file are wrapped in bare `catch {}` with no logging and no effect on the response. On a corrupt-JSON read, a Windows file lock (EBUSY/EPERM — a documented hazard in this repo), or a disk error, the patch is silently dropped but the endpoint returns success. The AI/persona believes its state mutation persisted when it did not, producing hard-to-diagnose state drift. This is the worst of the codebase's empty-catch sites because it is a write path on the request's primary side effect.
- **제안(검증반영)**: Two safe, independent steps. (1) Replace both bare catches with logged catches carrying session/file context, e.g. `catch (err) { console.error('[tools/'+name+'] variables write failed for '+sessionId+':', err) }` and the same per data file — this alone closes the diagnosability gap with zero behavioral change. (2) Wrap the two writes (and their read-modify) in await retryOnWindowsLock(() => { ... }) from src/lib/fs-retry.ts to absorb the documented EBUSY/EPERM/ENOTEMPTY from concurrent variables.json writers and the panel-engine watcher; the route is already async, so awaiting is free. Optionally collect a `failed: [fileName...]` array and add it to the success body (additive — verified all four callers tolerate extra keys) so the data-file patches, which have NO snapshot readback and are the truly silent case, become observable to the MCP client. Drop the 'flow through applyHookPatches' clause — that helper does not exist; if a future refactor extracts shared patch logic, apply the same logging/retry there. Do not change the HTTP status on partial write failure (keeps the optimistic UX in panel-action-registry intact).
- **위험**: Returning failures changes the response shape — callers (panel-action-registry client) must tolerate it. Logging-only is zero-risk and a safe first step.

### Style-check feeds RAW assistant content (choice JSON, image tokens, thinking) to the prose reviewer

- **차원**: RP 경험 · **위치**: `src/lib/session-instance.ts:888-902 (slice) + 1447-1460 (raw store) + 82-114/155-164 (existing strippers)` · **임팩트/노력(조정)**: medium/small · **확신**: high
